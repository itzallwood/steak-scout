# CLAUDE.md — Steak Scout Project

## Project Goal
Build an AI-powered agent that scrapes local grocery stores, meat markets, and butchers
to find the best deals on steaks. This is a learning project focused on understanding
web scraping fundamentals, AI agents, and LLM integration.

---

## Developer Context
- Python-comfortable developer, new to web scraping and AI agents
- Using Claude Code in VS Code integrated with GitHub
- Prioritize clear, readable, well-commented code over clever one-liners
- When introducing a new concept (e.g. a scraping pattern, agent tool, async behavior),
  add a short comment explaining WHY it works that way, not just what it does

---

## Stack

| Layer | Tool | Notes |
|---|---|---|
| Language | Python 3.11+ | Primary language throughout |
| Scraping (simple sites) | `requests` + `BeautifulSoup4` | For static HTML pages |
| Scraping (JS-heavy sites) | `Playwright` (async) | For sites that load prices via JavaScript |
| Agent framework | `LangGraph` | For orchestrating multi-step agent logic |
| LLM | Anthropic Claude via `anthropic` Python SDK | Sonnet model for cost efficiency |
| Database | `SQLite` via Python built-in `sqlite3` | Local file, no server needed |
| Output | Single-file HTML report | Generated by agent, opened in browser |
| Environment vars | `python-dotenv` + `.env` file | API keys and config, never committed to git |
| Testing | `pytest` | Basic tests for scrapers and db logic |
| Dependency management | `pip` + `requirements.txt` | Keep it simple |

## Target Stores (Phase 1)

| Store Type | Examples | Scraping Approach |
|---|---|---|
| National grocery chains | Kroger, Publix, Harris Teeter | Playwright (JS-heavy, may need careful handling) |
| Big box / warehouse | Walmart, Sam's Club, Costco | Playwright (heavily JS-rendered) |
| Local butchers / meat markets | Independent local shops | requests + BeautifulSoup (usually simpler sites) |

**Start with local butchers first** — simpler HTML, less bot detection, ideal for
learning scraping fundamentals before tackling the harder national chains.

## Architecture Overview

```
User runs agent
      │
      ▼
 price_agent.py  (LangGraph agent)
      │
      ├── Tool: scrape_store(store_name, url)
      │         └── Calls the right scraper class
      │                  ├── requests + BeautifulSoup  (simple sites)
      │                  └── Playwright                (JS-heavy sites)
      │
      ├── Tool: save_prices(store, cut, price)
      │         └── Writes to SQLite via db/queries.py
      │
      ├── Tool: get_best_deals(cut)
      │         └── Queries SQLite, returns ranked results
      │
      └── Tool: generate_report()
                └── report/generator.py → prices_report.html
```

The LLM (Claude) acts as the brain of the agent — it decides which tools to call,
in what order, and how to interpret results. LangGraph manages the state and flow
between tool calls.

## Key Conventions

### Scraper Design
- Every scraper inherits from `BaseScraper` in `base_scraper.py`
- Each scraper must implement a `scrape()` method that returns a list of dicts:
  ```python
  [{"store": "America's Choice Gourmet", "cut": "ribeye", "price": 40.00,
    "sale_price": 40.00, "original_price": 45.00, "url": "..."}]
  ```
- Never hardcode URLs in scrapers — pass them in or load from config
- Always include a `time.sleep()` between requests to be a polite scraper
- Log what you're doing at each step using Python's `logging` module

### Database
- Single SQLite file at `data/prices.db`
- Always store a `scraped_at` timestamp with every price record
- Never delete old records — this builds a price history over time

### Agent Tools
- Each LangGraph tool should do one thing only (single responsibility)
- Tools return structured dicts, not raw strings
- The agent should explain its reasoning in plain English before each tool call

### Environment Variables
Required in `.env` (see `.env.example`):
```
ANTHROPIC_API_KEY=your_key_here
```

### HTML Report
- Single self-contained `.html` file (no external dependencies)
- Generated fresh each run, saved to project root as `prices_report.html`
- Should show: best deal per cut, price comparison table, when data was last scraped
- Aim for a clean, readable design — this is a functional report, not a dashboard app

## What NOT to Do
- Don't scrape aggressively — always add delays between requests
- Don't commit `.env` or `data/prices.db` to GitHub
- Don't build the agent before the scrapers work — get data flowing first
- Don't use async everywhere from the start — introduce it only where Playwright requires it
- Don't over-engineer the database schema early — keep it simple and expand later
